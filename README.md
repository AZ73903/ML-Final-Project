Presentation Link

AAI201 Project

Problem Statement

LLM response quality improves when users provide personal context. Tools that help generate context files collect too much PII. The question: what kind of intake produces useful context without demanding sensitive information?

This matters because the people who would benefit most from better LLM interactions are the least equipped to write a good context file or judge what's safe to share.

This project compares two intake designs to see which produces embeddings with stronger structure for classifying worker domain, a first step to evaluting how well an LLM can classify a variety of question responses.

Data Source

Synthetic, generated by Claude with the assistance of Ollama. 400 synthetic personas across 7 domains, each answering two intake sets:

Set A is more direct, and preference-based questions, but I suspect not very engaging to actually complete Set B is behavioral open ended questions Each persona ran through 4 LLMs (gemma4, gpt-4o-mini, llama3.2, qwen2.5:14b) for 1,600 responses per set. Personas generated locally via Ollama responses via GPT-4o-mini on Azure. Dropped the "unknown" domain (7 samples). Final dataset: 1,593 samples across 6 balanced domains, embedded into 768-dim vectors. I had planned to use multiple models to respond to the question sets, but was unable to get enough compute quota in my azure account in the necessary time frame to really expand this project

Model Design

Input(768) → Dense(512, ReLU) → Dropout(0.2) → Dense(256, ReLU) → Dense(6, softmax Same MLP for both sets so the comparison is fair MLP fits because inputs are fixed-size dense embeddings, not sequences or images.

Evaluation Strategy

70/15/15 split. Accuracy as the primary metric since classes are balanced.

Set A test accuracy: 0.9833
Set B test accuracy: 0.9707
The gap is within normal variance. t-SNE showed Set A's domains broke into sub-clusters while Set B's stayed as single clusters, not driven by embedding model. Cause unresolved.

Bias & Fairness

The dataset is fully synthetic, so LLM stereotypes about occupations get baked into the personas. Domain balance was forced at generation Since my goal here is to address the questions, not the personas, bias was not particularily important to me here.

Overfitting Prevention

Dropout 0.2 after the first dense layer Early stopping on validation loss

Takeaways

6-way classification was too easy to separate the two intake methods on accuracy. I wish I had added more domains.
In regards to worker domain, there wasn't much difference in the model's capabaility to classify from either question set, a failure on my part in the question desing I believe.
I did automate further testing not included here on other features. Of particular note in my fake data test (take it with a grain of salt), the model could not predict comfort level with AI whatsoever, which I had though would have been more linked to worker domain.
Accuracy didnt really match what was visible in the embeddings
Synthetic personas inherit LLM biases real-world data needed to confirm
Like most things, I think I could make this much more meaningful with more questions, hindsight 20/20
